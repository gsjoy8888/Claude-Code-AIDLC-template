# 优先级：此工作流覆盖所有其他内置工作流
# 当用户请求软件开发时，始终首先遵循此工作流

## 自适应工作流原则
**工作流适应工作，而非相反。**

AI 模型根据以下因素智能评估所需阶段：
1. 用户陈述的意图和清晰度
2. 现有代码库状态（如果有）
3. 变更的复杂性和范围
4. 风险和影响评估

## 强制要求：规则详情加载
**关键**：执行任何阶段时，必须读取并使用规则详情文件中的相关内容。按顺序检查这些路径并使用第一个存在的：
- `.aidlc-rule-details/`（Cursor、Cline、Claude Code、GitHub Copilot）
- `.kiro/aws-aidlc-rule-details/`（Kiro IDE 和 CLI）
- `.amazonq/aws-aidlc-rule-details/`（Amazon Q Developer）

所有后续规则详情文件引用（例如 `common/process-overview.md`、`inception/workspace-detection.md`）都相对于上面解析的规则详情目录。

**通用规则**：在工作流开始时始终加载通用规则：
- 加载 `common/process-overview.md` 获取工作流概述
- 加载 `common/session-continuity.md` 获取会话恢复指导
- 加载 `common/content-validation.md` 获取内容验证要求
- 加载 `common/question-format-guide.md` 获取问题格式规则
- 在整个工作流执行过程中引用这些规则

## 强制要求：扩展加载
**关键**：在工作流开始时，递归扫描 `extensions/` 目录中的所有 `.md` 文件。这些是扩展规则文件，作为横切约束应用于整个工作流。

**加载过程**：
1. 列出 `extensions/` 下的所有子目录（例如 `extensions/security/`、`extensions/compliance/`）
2. 加载这些子目录中找到的每个 `.md` 文件
3. 每个扩展文件定义其自己的验证标准和执行规则作为横切约束

**执行**：
- 扩展规则是硬约束，不是可选指导
- 在每个阶段，模型根据阶段目的、生成的工件和工作上下文智能评估哪些扩展规则适用——仅执行相关规则
- 不适用于当前阶段的规则应在合规性摘要中标记为 N/A（这不是阻塞性发现）
- 不符合任何适用的已启用扩展规则是**阻塞性发现**——在解决之前不要呈现阶段完成
- 呈现阶段完成时，包括扩展规则合规性摘要（每条规则的合规/不合规/N/A，以及 N/A 判定的简要理由）

**条件执行**：扩展可以有条件地启用/禁用。参见 `inception/requirements-analysis.md` 了解收集机制。在任何阶段执行任何扩展之前，检查其在 `aidlc-docs/aidlc-state.md` 的 `## Extension Configuration` 下的 `Enabled` 状态。跳过已禁用的扩展并在 audit.md 中记录跳过。如果不存在配置则默认执行。没有 `## Applicability Question` 的扩展始终执行。

## 强制要求：内容验证
**关键**：在创建任何文件之前，必须根据 `common/content-validation.md` 规则验证内容：
- 验证 Mermaid 图表语法
- 验证 ASCII 艺术图表（参见 `common/ascii-diagram-standards.md`）
- 正确转义特殊字符
- 为复杂视觉内容提供文本替代
- 测试内容解析兼容性

## 强制要求：问题文件格式
**关键**：在任何阶段提问时，必须遵循问题格式指南。

**参见 `common/question-format-guide.md` 获取完整的问题格式规则，包括**：
- 多选格式（A、B、C、D、E 选项）
- [Answer]: 标签使用
- 答案验证和歧义解决

## 强制要求：自定义欢迎消息
**关键**：开始任何软件开发请求时，必须显示欢迎消息。

**如何显示欢迎消息**：
1. 从 `common/welcome-message.md`（在已解析的规则详情目录中）加载欢迎消息
2. 向用户显示完整消息
3. 这应该仅在新工作流开始时执行一次
4. 不要在后续交互中加载此文件以节省上下文空间

# 启动阶段

**目的**：规划、需求收集和架构决策

**重点**：确定构建什么以及为什么

**启动阶段的阶段**：
- 工作空间检测（始终）
- 逆向工程（条件性 - 仅棕地）
- 需求分析（始终 - 自适应深度）
- 用户故事（条件性）
- 工作流规划（始终）
- 应用设计（条件性）
- 单元生成（条件性）

---

## 工作空间检测（始终执行）

1. **强制要求**：在 audit.md 中记录初始用户请求及完整原始输入
2. 从 `inception/workspace-detection.md` 加载所有步骤
3. 执行工作空间检测：
   - 检查现有 aidlc-state.md（如果找到则恢复）
   - 扫描工作空间中的现有代码
   - 确定是棕地还是绿地
   - 检查现有逆向工程工件
4. 确定下一阶段：逆向工程（如果是棕地且没有工件）或需求分析
5. **强制要求**：在 audit.md 中记录发现
6. 向用户呈现完成消息（参见 workspace-detection.md 了解消息格式）
7. 自动进入下一阶段

## 逆向工程（条件性 - 仅棕地）

**执行条件**：
- 检测到现有代码库
- 未找到先前的逆向工程工件

**跳过条件**：
- 绿地项目
- 存在先前的逆向工程工件

**执行**：
1. **强制要求**：在 audit.md 中记录逆向工程开始
2. 从 `inception/reverse-engineering.md` 加载所有步骤
3. 执行逆向工程：
   - 分析所有包和组件
   - 生成涵盖业务事务的整个系统的业务概述
   - 生成架构文档
   - 生成代码结构文档
   - 生成 API 文档
   - 生成组件清单
   - 生成描述业务事务如何跨组件实现的交互图
   - 生成技术栈文档
   - 生成依赖关系文档

4. **等待明确批准**：呈现详细完成消息（参见 reverse-engineering.md 了解消息格式）- 在用户确认之前不要继续
5. **强制要求**：在 audit.md 中记录用户的响应及完整原始输入


## 需求分析（始终执行 - 自适应深度）

**始终执行**，但深度根据请求清晰度和复杂性而变化：
- **最小**：简单、清晰的请求 - 仅记录意图分析
- **标准**：正常复杂度 - 收集功能和非功能需求
- **全面**：复杂、高风险 - 具有可追溯性的详细需求

**执行**：
1. **强制要求**：在 audit.md 中记录此阶段的任何用户输入
2. 从 `inception/requirements-analysis.md` 加载所有步骤
3. 执行需求分析：
   - 加载逆向工程工件（如果是棕地）
   - 分析用户请求（意图分析）
   - 确定所需的需求深度
   - 评估当前需求
   - 提出澄清问题（如果需要）
   - 生成需求文档
4. 以适当深度执行（最小/标准/全面）
5. **等待明确批准**：遵循 requirements-analysis.md 详细步骤中的批准格式 - 在用户确认之前不要继续
6. **强制要求**：在 audit.md 中记录用户的响应及完整原始输入

## 用户故事（条件性）

**智能评估**：使用多因素分析确定用户故事是否增加价值：

**始终执行条件**（高优先级指标）：
- 新的面向用户的功能或特性
- 影响用户工作流或交互的变更
- 涉及多个用户类型或角色
- 具有验收标准需求的复杂业务需求
- 需要跨职能团队协作
- 面向客户的 API 或服务变更
- 新产品能力或增强功能

**可能执行条件**（中优先级 - 评估复杂性）：
- 对现有面向用户功能的修改
- 间接影响用户体验的后端变更
- 影响用户工作流的集成工作
- 具有用户可见收益的性能改进
- 影响用户交互的安全增强
- 影响用户数据或报告的数据模型变更

**基于复杂性的评估**：对于中优先级情况，如果满足以下条件则执行用户故事：
- 请求涉及多个组件或服务
- 变更跨越多个用户接触点
- 业务逻辑复杂或有多个场景
- 需求存在故事可以澄清的歧义
- 实现影响多个用户旅程
- 变更具有重大业务影响或风险

**仅跳过条件**（低优先级 - 简单情况）：
- 纯内部重构，对用户零影响
- 具有清晰、隔离范围的简单错误修复
- 没有面向用户影响的基础设施变更
- 没有功能变更的技术债务清理
- 开发者工具或构建过程改进
- 仅文档更新

**评估标准**：如有疑问，在以下情况下倾向于包含用户故事：
- 涉及业务利益相关者的请求
- 需要用户验收测试的变更
- 具有多种实现方法的功能
- 受益于团队共同理解的工作
- 需求清晰度有价值的项目

**评估过程**：
1. 分析请求复杂性和范围
2. 识别用户影响（直接或间接）
3. 评估业务上下文和利益相关者需求
4. 考虑团队协作收益
5. 对于边界情况默认包含

**注意**：如果执行了需求分析，故事可以引用并基于这些需求构建。

**用户故事在一个阶段内有两个部分**：
1. **第 1 部分 - 规划**：创建包含问题的故事计划，收集答案，分析歧义，获得批准
2. **第 2 部分 - 生成**：执行已批准的计划以生成故事和角色

**执行**：
1. **强制要求**：在 audit.md 中记录此阶段的任何用户输入
2. 从 `inception/user-stories.md` 加载所有步骤
3. **强制要求**：执行智能评估（user-stories.md 中的步骤 1）以验证是否需要用户故事
4. 加载逆向工程工件（如果是棕地）
5. 如果存在需求，在创建故事时引用它们
6. 以适当深度执行（最小/标准/全面）
7. **第 1 部分 - 规划**：创建包含问题的故事计划，等待用户答案，分析歧义，获得批准
8. **第 2 部分 - 生成**：执行已批准的计划以生成故事和角色
9. **等待明确批准**：遵循 user-stories.md 详细步骤中的批准格式 - 在用户确认之前不要继续
10. **强制要求**：在 audit.md 中记录用户的响应及完整原始输入

## 工作流规划（始终执行）

# 🟢 构建阶段

**目的**：详细设计、NFR 实现和代码生成

**重点**：确定如何构建

**构建阶段的阶段**：
- 每单元循环（为每个单元执行）：
  - 功能设计（条件性，每单元）
  - NFR 需求（条件性，每单元）
  - NFR 设计（条件性，每单元）
  - 基础设施设计（条件性，每单元）
  - 代码生成（始终，每单元）
- 构建和测试（始终 - 所有单元完成后）

**注意**：每个单元在移至下一个单元之前完全完成（设计 + 代码）。

---

## 每单元循环（为每个单元执行）

**对于每个工作单元，按顺序执行以下阶段：**

### 功能设计（条件性，每单元）

**执行条件**：
- 新数据模型或模式
- 复杂业务逻辑
- 业务规则需要详细设计

**跳过条件**：
- 简单逻辑变更
- 没有新业务逻辑

**执行**：
1. **强制要求**：在 audit.md 中记录此阶段的任何用户输入
2. 从 `construction/functional-design.md` 加载所有步骤
3. 为此单元执行功能设计
4. **强制要求**：呈现 functional-design.md 中定义的标准化 2 选项完成消息 - 不要使用新兴的 3 选项行为
5. **等待明确批准**：用户必须在"请求变更"或"继续下一阶段"之间选择 - 在用户确认之前不要继续
6. **强制要求**：在 audit.md 中记录用户的响应及完整原始输入

## 关键原则

- **自适应执行**：仅执行增加价值的阶段
- **透明规划**：在开始前始终显示执行计划
- **用户控制**：用户可以请求包含/排除阶段
- **进度跟踪**：使用已执行和跳过的阶段更新 aidlc-state.md
- **完整审计跟踪**：在 audit.md 中记录所有用户输入和 AI 响应及时间戳
  - **关键**：完全按照提供的方式捕获用户的完整原始输入
  - **关键**：永远不要在审计日志中总结或改述用户输入
  - **关键**：记录每次交互，不仅仅是批准
- **质量关注**：复杂变更获得完整处理，简单变更保持高效
- **内容验证**：在文件创建前始终根据 content-validation.md 规则验证内容
- **无新兴行为**：构建阶段必须使用其各自规则文件中定义的标准化 2 选项完成消息。不要创建 3 选项菜单或其他新兴导航模式。

## 强制要求：计划级复选框执行

### 计划执行的强制规则
1. **永远不要在不更新计划复选框的情况下完成任何工作**
2. **在完成计划文件中描述的任何步骤后立即标记该步骤 [x]**
3. **这必须在完成工作的同一交互中发生**
4. **无例外**：每个计划步骤完成必须通过复选框更新进行跟踪

### 两级复选框跟踪系统
- **计划级**：跟踪每个阶段内的详细执行进度
- **阶段级**：跟踪 aidlc-state.md 中的整体工作流进度
- **立即更新**：在完成工作的同一交互中进行所有进度更新

## 提示记录要求
- **强制要求**：在 audit.md 中记录每个用户输入（提示、问题、响应）及时间戳
- **强制要求**：完全按照提供的方式捕获用户的完整原始输入（永远不要总结）
- **强制要求**：在询问用户之前记录每个批准提示及时间戳
- **强制要求**：在收到后记录每个用户响应及时间戳
- **关键**：始终追加更改以编辑 audit.md 文件，永远不要使用完全覆盖其内容的工具和命令
- **关键**：使用覆盖整个 audit.md 内容并导致重复的文件写入工具和命令
- 使用 ISO 8601 格式的时间戳（YYYY-MM-DDTHH:MM:SSZ）
- 为每个条目包含阶段上下文

### audit.md 的正确工具使用

❌ 错误：
