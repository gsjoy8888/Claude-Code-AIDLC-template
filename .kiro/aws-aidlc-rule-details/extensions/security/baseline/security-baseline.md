# 基线安全规则

## 概述
这些安全规则是适用于所有 AI-DLC 阶段的强制性横切约束。它们不是可选指导——而是各阶段在生成问题、产出设计制品、生成代码和呈现完成消息时必须执行的硬性约束。

**执行要求**:在每个适用阶段,模型必须在向用户呈现阶段完成消息之前验证对这些规则的合规性。

### 阻塞性安全发现行为
**阻塞性安全发现**意味着:
1. 该发现必须在阶段完成消息的"安全发现"部分中列出,包含 SECURITY 规则 ID 和描述
2. 在所有阻塞性发现解决之前,该阶段不得呈现"继续到下一阶段"选项
3. 模型必须仅呈现"请求变更"选项,并清楚说明需要变更的内容
4. 该发现必须记录在 `aidlc-docs/audit.md` 中,包含 SECURITY 规则 ID、描述和阶段上下文

如果某个 SECURITY 规则不适用于当前项目(例如,当不存在数据存储时的 SECURITY-01),在合规性摘要中将其标记为 **N/A**——这不是阻塞性发现。

### 默认执行
本文档中的所有规则默认都是**阻塞性的**。如果任何规则的验证标准未满足,即为阻塞性安全发现——遵循上述定义的阻塞性发现行为。

### 验证标准格式
本文档中的验证项是描述合规性检查的普通项目符号。它们与阶段计划文件中使用的 `- [ ]` / `- [x]` 进度跟踪复选框不同。每个项目应在审查期间评估为合规或不合规。

## 规则 SECURITY-01:静态和传输加密

**规则**:每个数据持久化存储(数据库、对象存储、文件系统、缓存或任何等效物)必须具有:
- 使用托管密钥服务或客户管理密钥启用静态加密
- 强制传输加密(所有进出存储的数据移动使用 TLS 1.2+)

**验证**:
- 没有存储资源在未配置加密配置块的情况下定义
- 没有数据库连接字符串使用未加密协议
- 对象存储强制执行静态加密并通过策略拒绝非 TLS 请求
- 数据库实例已启用存储加密并强制执行 TLS 连接

---

## 规则 SECURITY-02:网络中介的访问日志

**规则**:每个处理外部流量的面向网络的中介必须启用访问日志。这包括:
- 负载均衡器 → 访问日志到持久化存储
- API 网关 → 执行日志和访问日志到集中式日志服务
- CDN 分发 → 标准日志或实时日志

**验证**:
- 没有负载均衡器资源在未启用访问日志的情况下定义
- 没有 API 网关阶段在未配置访问日志的情况下定义
- 没有 CDN 分发在未配置日志的情况下定义

---

## 规则 SECURITY-03:应用程序级日志

**规则**:每个部署的应用程序组件必须包含结构化日志基础设施:
- 必须配置日志框架
- 日志输出必须定向到集中式日志服务
- 日志必须包含:时间戳、关联/请求 ID、日志级别和消息
- 敏感数据(密码、令牌、PII)不得出现在日志输出中

**验证**:
- 每个服务/函数入口点都包含已配置的日志记录器
- 生产代码中没有使用临时日志语句作为主要日志机制
- 日志配置将输出路由到集中式日志服务
- 没有记录密钥、令牌或 PII

---

## 规则 SECURITY-04:Web 应用程序的 HTTP 安全标头

**规则**:必须在所有提供 HTML 的端点上设置以下 HTTP 响应标头:

| 标头 | 必需值 |
|---|---|
| `Content-Security-Policy` | 定义限制性策略(至少:`default-src 'self'`) |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` |
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY`(或如果需要框架则为 `SAMEORIGIN`) |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |

**注意**:`X-XSS-Protection` 在现代浏览器中已弃用。请改用 `Content-Security-Policy`。

**验证**:
- 中间件或响应拦截器设置所有必需的标头
- CSP 策略不使用 `unsafe-inline` 或 `unsafe-eval`,除非有文档化的理由
- HSTS max-age 至少为 31536000(1 年)

---

## 规则 SECURITY-05:所有 API 参数的输入验证

**规则**:每个 API 端点(REST、GraphQL、gRPC、WebSocket)必须在处理之前验证所有输入参数。验证必须包括:
- **类型检查**:拒绝意外类型
- **长度/大小边界**:对字符串强制执行最大长度,对数组和有效负载强制执行最大大小
- **格式验证**:对结构化输入(电子邮件、日期、ID)使用白名单(正则表达式或模式)
- **清理**:转义或拒绝用户提供字符串中的 HTML/脚本内容以防止 XSS
- **注入防护**:对所有数据库操作使用参数化查询(绝不使用字符串拼接)

**验证**:
- 每个 API 处理程序使用验证库或模式
- 没有原始用户输入被拼接到 SQL、NoSQL 或 OS 命令中
- 字符串输入具有明确的最大长度约束
- 在框架或网关级别配置请求正文大小限制

---

## 规则 SECURITY-06:最小权限访问策略

**规则**:每个身份和访问管理策略、角色或权限边界必须遵循最小权限:
- 使用特定资源标识符——绝不使用通配符资源,除非 API 不支持资源级权限(记录例外)
- 使用特定操作——绝不使用通配符操作
- 尽可能限定条件范围
- 将读取和写入权限分离到不同的策略语句中

**验证**:
- 没有策略包含通配符操作或通配符资源,除非有文档化的例外
- 没有服务角色具有比服务实际调用更广泛的权限
- 尽可能避免内联策略,优先使用托管策略
- 每个角色都有限定到特定服务或账户的信任策略

---

## 规则 SECURITY-07:限制性网络配置

**规则**:所有网络配置(安全组、网络 ACL、路由表)必须遵循默认拒绝:
- 防火墙规则:仅打开应用程序所需的特定端口
- 除了端口 80/443 上的面向公众的负载均衡器外,没有源为 `0.0.0.0/0` 的入站规则
- 除非明确证明合理,否则没有所有端口上目标为 `0.0.0.0/0` 的出站规则
- 私有子网不得具有直接的互联网网关路由
- 在可用的情况下使用私有端点进行云服务访问

**验证**:
- 除了公共负载均衡器上的端口 80/443 外,没有防火墙规则允许任何端口上的入站 `0.0.0.0/0`
- 数据库和应用程序防火墙规则将源限制为特定 CIDR 块或安全组引用
- 私有子网通过 NAT 网关路由(而不是互联网网关)
- 对高流量云服务调用使用私有端点

---

## 规则 SECURITY-08:应用程序级访问控制

**规则**:每个访问或修改资源的应用程序端点必须在应用程序层强制执行授权检查:
- **默认拒绝**:所有路由/端点必须要求身份验证,除非明确标记为公共
- **对象级授权**:每个通过 ID 引用资源的请求必须验证请求用户/主体拥有或有权访问该资源(防止 IDOR)
- **功能级授权**:管理或特权操作必须在服务器端检查调用者的角色/权限——绝不依赖客户端隐藏
- **CORS 策略**:跨源资源共享必须限制为明确允许的源——绝不在经过身份验证的端点上使用 `Access-Control-Allow-Origin: *`
- **令牌验证**:JWT 或会话令牌必须在每个请求上在服务器端验证(签名、过期、受众、颁发者)

**验证**:
- 每个控制器/处理程序都应用了授权中间件或守卫
- 没有端点在未验证调用者的所有权或权限的情况下返回资源 ID 的数据
- 管理/特权路由在服务器端强制执行明确的角色检查
- CORS 配置不在经过身份验证的端点上使用通配符源
- 令牌验证在每个请求上在服务器端进行(而不仅仅是在登录时)

---

## 规则 SECURITY-09:安全加固和错误配置防护

**规则**:所有部署的组件必须遵循加固基线:
- **无默认凭据**:默认用户名/密码必须在部署前更改或禁用
- **最小安装**:删除或禁用未使用的功能、框架、示例应用程序和文档端点
- **错误处理**:生产错误响应不得向最终用户暴露堆栈跟踪、内部路径、框架版本或数据库详细信息
- **目录列表**:Web 服务器必须禁用目录列表
- **云存储**:云对象存储必须阻止公共访问,除非明确需要并记录
- **补丁管理**:运行时环境、框架和 OS 镜像必须使用当前支持的版本

**验证**:
- 配置文件、环境变量或 IaC 模板中不存在默认凭据
- 生产中的错误响应返回通用消息(无堆栈跟踪或内部详细信息)
- 云对象存储已阻止公共访问,除非存在文档化的例外
- 没有部署示例/演示应用程序或默认页面
- 框架和运行时版本是当前且受支持的



---

## 规则 SECURITY-10:软件供应链安全

**规则**:每个项目必须管理其软件供应链:
- **依赖项固定**:所有依赖项必须使用精确版本或锁定文件
- **漏洞扫描**:必须配置依赖项漏洞扫描器
- **无未使用的依赖项**:删除未主动使用的包
- **仅受信任的源**:依赖项必须从官方注册表或经过验证的私有注册表拉取——不使用未经审查的第三方源
- **SBOM**:项目必须为生产部署生成软件物料清单
- **CI/CD 完整性**:构建管道必须使用固定的工具版本和经过验证的基础镜像——生产 Dockerfile 或 CI 配置中不使用 `latest` 标签

**验证**:
- 存在锁定文件并已提交到版本控制
- CI/CD 中包含依赖项漏洞扫描步骤或在构建说明中记录
- 不包含未使用或已废弃的依赖项
- Dockerfile 和 CI 配置不对生产使用 `latest` 或未固定的镜像标签
- 依赖项来自官方或经过验证的注册表

## 规则 SECURITY-11:安全设计原则

**规则**:应用程序设计必须从一开始就纳入安全性:
- **关注点分离**:安全关键逻辑(身份验证、授权、支付处理)必须隔离在专用模块中——不分散在代码库中
- **纵深防御**:没有单一控制应该是唯一的防线——分层控制(验证 + 授权 + 加密)
- **速率限制**:面向公众的端点必须实施速率限制或节流以防止滥用
- **业务逻辑滥用**:设计必须考虑误用案例——不仅仅是正常路径场景

**验证**:
- 安全关键逻辑封装在专用模块或服务中
- 在面向公众的 API 上配置速率限制
- 设计文档至少解决一个误用/滥用场景

---

## 规则 SECURITY-12:身份验证和凭据管理

**规则**:每个具有用户身份验证的应用程序必须实施:
- **密码策略**:最少 8 个字符,对照泄露密码列表检查
- **凭据存储**:密码必须使用自适应算法进行哈希——绝不使用弱或非自适应哈希
- **多因素身份验证**:管理账户必须支持 MFA,所有用户应该可以使用 MFA
- **会话管理**:会话必须具有服务器端过期,在注销时失效,并使用 secure/httpOnly/sameSite cookie 属性
- **暴力破解保护**:登录端点必须在重复失败后实施账户锁定、渐进延迟或 CAPTCHA
- **无硬编码凭据**:源代码或 IaC 模板中不得有密码、API 密钥或密钥——使用密钥管理器

**验证**:
- 密码哈希使用自适应算法(不使用弱或非自适应哈希)
- 会话 cookie 设置 `Secure`、`HttpOnly` 和 `SameSite` 属性
- 登录端点具有暴力破解保护(锁定、延迟或 CAPTCHA)
- 源代码或配置文件中没有硬编码凭据
- 管理账户支持 MFA
- 会话在注销时失效并具有定义的过期时间

---

## 规则 SECURITY-13:软件和数据完整性验证

**规则**:系统必须验证软件和数据的完整性:
- **反序列化安全**:不受信任的数据不得在未验证的情况下反序列化——使用安全的反序列化库或允许类型的白名单
- **制品完整性**:下载的依赖项、插件和更新必须通过校验和或数字签名进行验证
- **CI/CD 管道安全**:构建管道必须限制谁可以修改管道定义——将代码作者和部署批准者的职责分开
- **CDN 和外部资源**:从外部 CDN 加载的脚本或资源必须使用子资源完整性(SRI)哈希
- **数据完整性**:关键数据修改必须可审计(谁在何时更改了什么)

**验证**:
- 没有不受信任输入的不安全反序列化
- 从 CDN 加载的外部脚本包含 SRI 完整性属性
- CI/CD 管道定义受访问控制且更改可审计
- 关键数据更改记录了操作者、时间戳和前后值

---

## 规则 SECURITY-14:告警和监控

**规则**:除了日志记录(SECURITY-02、SECURITY-03)外,系统必须包括:
- **安全事件告警**:必须为高价值安全事件配置告警:重复身份验证失败、权限提升尝试、来自异常位置的访问和授权失败
- **日志完整性**:日志必须存储在仅追加或防篡改存储中——应用程序代码不得能够删除或修改其自己的审计日志
- **日志保留**:日志必须保留适合应用程序合规要求的最短期限(默认:最少 90 天)
- **监控仪表板**:必须为关键运营和安全指标定义监控仪表板或告警配置

**验证**:
- 为身份验证失败和授权违规配置告警
- 应用程序日志组设置了保留策略(最少 90 天)
- 应用程序角色没有删除其自己的日志组/流的权限
- 安全相关事件(登录失败、访问被拒绝、权限更改)生成告警

---

## 规则 SECURITY-15:异常处理和故障安全默认值

**规则**:每个应用程序必须安全地处理异常情况:
- **捕获和处理**:所有外部调用(数据库、API、文件 I/O)必须具有明确的错误处理——生产中不得有未处理的 promise 拒绝或未捕获的异常
- **故障关闭**:出错时,系统必须拒绝访问或停止操作——绝不故障开放
- **资源清理**:错误路径必须释放资源(连接、文件句柄、锁)——使用 try/finally、using 语句或等效模式
- **面向用户的错误**:向用户显示的错误消息必须是通用的——没有内部详细信息或系统信息
- **全局错误处理程序**:应用程序必须具有全局/顶级错误处理程序,捕获未处理的异常,记录它们(根据 SECURITY-03),并返回安全响应

**验证**:
- 所有外部调用(DB、HTTP、文件 I/O)都有明确的错误处理(try/catch、.catch()、错误回调)
- 在应用程序入口点配置全局错误处理程序
- 错误路径不绕过授权或验证检查(故障关闭)
- 在错误路径中清理资源(关闭连接、回滚事务)
- 应用程序代码中没有未处理的 promise 拒绝或未捕获的异常警告

---

## 执行集成

这些规则是适用于每个 AI-DLC 阶段的横切约束。在每个阶段:
- 根据产出的制品评估所有 SECURITY 规则验证标准
- 在阶段完成摘要中包含"安全合规性"部分,将每个规则列为合规、不合规或 N/A
- 如果任何规则不合规,这是阻塞性安全发现——遵循概述中定义的阻塞性发现行为
- 在设计文档和测试说明中包含安全规则引用
