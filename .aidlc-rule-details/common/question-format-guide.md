# 问题格式指南

## 强制要求：所有问题必须使用此格式

### 规则：永远不要在聊天中提问
**关键要求**：您绝不能直接在聊天中提问。所有问题必须放在专用问题文件中。

### 问题文件格式

#### 文件命名约定
- 使用描述性名称：`{phase-name}-questions.md`
- 示例：
  - `classification-questions.md`
  - `requirements-questions.md`
  - `story-planning-questions.md`
  - `design-questions.md`

#### 问题结构
每个问题必须包含有意义的选项加上"其他"作为最后一个选项：

```markdown
## 问题 [编号]
[清晰、具体的问题文本]

A) [第一个有意义的选项]
B) [第二个有意义的选项]
[...根据需要添加其他选项...]
X) 其他（请在下面的 [Answer]: 标签后描述）

[Answer]: 
```

**关键要求**：
- "其他"对于每个问题都是强制性的，作为最后一个选项
- 仅包含有意义的选项 - 不要编造选项来填充位置
- 使用尽可能多或尽可能少的选项（最少 2 + 其他）

### 完整示例

```markdown
# 需求澄清问题

请回答以下问题以帮助澄清需求。

## 问题 1
主要的用户身份验证方法是什么？

A) 用户名和密码
B) 社交媒体登录（Google、Facebook）
C) 单点登录（SSO）
D) 多因素身份验证
E) 其他（请在下面的 [Answer]: 标签后描述）

[Answer]: 

## 问题 2
这将是 Web 还是移动应用程序？

A) Web 应用程序
B) 移动应用程序
C) Web 和移动都有
D) 其他（请在下面的 [Answer]: 标签后描述）

[Answer]: 

## 问题 3
这是新项目还是现有代码库？

A) 新项目（绿地）
B) 现有代码库（棕地）
C) 其他（请在下面的 [Answer]: 标签后描述）

[Answer]: 
```

### 用户响应格式
用户将通过在 [Answer]: 标签后填写字母选择来回答：

```markdown
## 问题 1
主要的用户身份验证方法是什么？

A) 用户名和密码
B) 社交媒体登录（Google、Facebook）
C) 单点登录（SSO）
D) 多因素身份验证

[Answer]: C
```

### 读取用户响应
用户确认完成后：
1. 读取问题文件
2. 提取 [Answer]: 标签后的答案
3. 验证所有问题都已回答
4. 根据响应继续分析

### 多项选择指南

#### 选项数量
- 最少：2 个有意义的选项 + "其他"（A、B、C）
- 典型：3-4 个有意义的选项 + "其他"（A、B、C、D、E）
- 最多：5 个有意义的选项 + "其他"（A、B、C、D、E、F）
- **关键要求**：不要仅为填充位置而编造选项 - 仅包含有意义的选择

#### 选项质量
- 使选项互斥
- 涵盖最常见的场景
- 仅包含有意义、现实的选项
- **始终将"其他"作为最后一个选项**（强制性）
- 具体明确
- **不要编造选项来填充 A、B、C、D 位置**

#### 好的示例：
```markdown
## 问题 5
将使用什么数据库技术？

A) 关系型（PostgreSQL、MySQL）
B) NoSQL 文档（MongoDB、DynamoDB）
C) NoSQL 键值（Redis、Memcached）
D) 图数据库（Neo4j、Neptune）
E) 其他（请在下面的 [Answer]: 标签后描述）

[Answer]: 
```

#### 不好的示例（避免）：
```markdown
## 问题 5
您将使用什么数据库？

A) 是
B) 否
C) 也许

[Answer]: 
```

### 工作流集成

#### 步骤 1：创建问题文件
```markdown
创建 aidlc-docs/{phase-name}-questions.md 包含所有问题
```

#### 步骤 2：通知用户
```
"我已创建 {phase-name}-questions.md，包含 [X] 个问题。
请通过在 [Answer]: 标签后填写字母选择来回答每个问题。
如果没有选项符合您的需求，请选择最后一个选项（其他）并描述您的偏好。完成后告诉我。"
```

#### 步骤 3：等待确认
等待用户说"完成"、"已完成"、"已结束"或类似的话。

#### 步骤 4：读取和分析
```
读取 aidlc-docs/{phase-name}-questions.md
提取所有答案
验证完整性
继续分析
```

### 错误处理

#### 缺少答案
如果任何 [Answer]: 标签为空：
```
"我注意到问题 [X] 没有回答。在继续之前，请使用字母选择为所有问题提供答案。"
```

#### 无效答案
如果答案不是有效的字母选择：
```
"问题 [X] 的答案 '[answer]' 无效。
请仅使用问题中提供的字母选择。"
```

#### 模糊答案
如果用户提供解释而不是字母：
```
"对于问题 [X]，请提供最符合您答案的字母选择。
如果没有匹配的，请选择'其他'并在 [Answer]: 标签后添加您的描述。"
```

### 矛盾和模糊性检测

**强制要求**：读取用户响应后，您必须检查矛盾和模糊性。

#### 检测矛盾
寻找逻辑上不一致的答案：
- 范围不匹配："Bug 修复"但"整个代码库受影响"
- 风险不匹配："低风险"但"破坏性更改"
- 时间线不匹配："快速修复"但"多个子系统"
- 影响不匹配："单个组件"但"重大架构更改"

#### 检测模糊性
寻找不清楚或边界响应：
- 可能适合多个分类的答案
- 缺乏具体性的响应
- 多个问题之间的冲突指标

#### 创建澄清问题
如果检测到矛盾或模糊性：

1. **创建澄清文件**：`{phase-name}-clarification-questions.md`
2. **解释问题**：清楚地说明检测到什么矛盾/模糊性
3. **提出针对性问题**：使用多项选择格式解决问题
4. **引用原始问题**：显示哪些问题有冲突的答案

**示例**：
```markdown
# [阶段名称] 澄清问题

我在您的响应中检测到需要澄清的矛盾：

## 矛盾 1：[简要描述]
您指出了"[答案 A]"（Q[X]:[字母]）但也指出了"[答案 B]"（Q[Y]:[字母]）。
这些响应是矛盾的，因为 [解释]。

### 澄清问题 1
[解决矛盾的具体问题]

A) [解决第一个答案的选项]
B) [解决第二个答案的选项]
C) [提供中间立场的选项]
D) [重新构建问题的选项]

[Answer]: 

## 模糊性 1：[简要描述]
您对 Q[X] 的响应（"[答案]"）是模糊的，因为 [解释]。

### 澄清问题 2
[澄清模糊性的具体问题]

A) [清晰选项 1]
B) [清晰选项 2]
C) [清晰选项 3]
D) [清晰选项 4]

[Answer]: 
```

#### 澄清工作流

1. **检测**：分析所有响应以查找矛盾/模糊性
2. **创建**：如果发现问题，生成澄清问题文件
3. **通知**：告诉用户问题和澄清文件
4. **等待**：在用户提供澄清之前不要继续
5. **重新验证**：澄清后，再次检查一致性
6. **继续**：仅在所有矛盾得到解决后才前进

#### 示例用户消息
```
"我在您的响应中检测到 2 个矛盾：

1. Bug 修复范围 vs. 代码库影响（Q1 vs Q2）
2. 低风险 vs. 破坏性更改（Q7 vs Q4）

我已创建 classification-clarification-questions.md，包含 2 个问题来解决这些问题。
在我继续分类之前，请回答这些澄清问题。"
```

### 最佳实践

1. **具体**：问题应清晰明确
2. **全面**：涵盖所有必要信息
3. **简洁**：保持问题专注于一个主题
4. **实用**：选项应现实且可操作
5. **一致**：在所有问题文件中使用相同格式
