# 工作流中期更改和阶段管理

## 概述

用户可能在工作流期间请求更改执行计划或阶段执行。本文档提供有关安全有效地处理这些请求的指导。

---

## 工作流中期更改的类型

### 1. 添加跳过的阶段

**场景**：用户想要添加最初跳过的阶段

**示例**："实际上，即使我们跳过了该步骤，我也想添加用户故事"

**处理**：
1. **确认请求**："您想添加用户故事步骤。这将创建用户故事和角色。确认？"
2. **检查依赖项**：验证所有先决条件阶段是否完成
3. **更新执行计划**：将阶段添加到 `execution-plan.md` 并附上理由
4. **更新状态**：在 `aidlc-state.md` 中将阶段标记为"待处理"
5. **执行阶段**：遵循正常阶段执行过程
6. **记录更改**：在 `audit.md` 中记录时间戳和原因

**考虑因素**：
- 可能需要更新可以从新产物中受益的后续阶段
- 现有产物可能需要修订以整合新信息
- 时间线将延长

---

### 2. 跳过计划的阶段

**场景**：用户想要跳过计划执行的阶段

**示例**："现在让我们跳过 NFR 设计步骤"

**处理**：
1. **确认请求**："您想跳过 NFR 设计。这意味着不会整合 NFR 模式或逻辑组件。确认？"
2. **警告影响**：解释将缺少什么以及潜在后果
3. **获得明确确认**：用户必须明确确认理解影响
4. **更新执行计划**：将阶段标记为"已跳过"并附上原因
5. **更新状态**：在 `aidlc-state.md` 中将阶段标记为"已跳过"
6. **调整后续阶段**：注意后续阶段可能需要手动设置
7. **记录更改**：在 `audit.md` 中记录时间戳和原因

**考虑因素**：
- 后续阶段可能失败或需要手动干预
- 用户接受缺少产物的责任
- 如果需要，可以稍后添加回来

---

### 3. 重新启动当前步骤

**场景**：用户对当前步骤结果不满意并想重做

**示例**："我不喜欢这些用户故事。我们可以重新开始吗？"

**处理**：
1. **了解关注点**："您想更改故事的哪些具体内容？"
2. **提供选项**：
   - **选项 A**：修改现有产物（更快，保留一些工作）
   - **选项 B**：完全重新启动（全新开始，更多时间）
3. **如果选择重新启动**：
   - 归档现有产物：`{artifact}.backup.{timestamp}`
   - 重置计划文件中的步骤复选框
   - 在 `aidlc-state.md` 中将步骤标记为"进行中"
   - 清除步骤完成状态
   - 从头重新执行
4. **记录更改**：记录重新启动的原因以及将更改什么

**考虑因素**：
- 现有工作将丢失（但已备份）
- 可能需要重做依赖步骤
- 时间线将延长

---

### 4. 重新启动先前步骤

**场景**：用户想要返回并重做已完成的步骤

**示例**："我想更改我们之前做出的架构决策"

**处理**：
1. **评估影响**：识别依赖于要重新启动的步骤的所有步骤
2. **警告用户**："重新启动应用设计将需要重做：单元规划、单元生成、每单元设计（所有单元）、代码规划、代码生成。确认？"
3. **获得明确确认**：用户必须理解完整影响
4. **如果确认**：
   - 归档所有受影响的产物
   - 在 `aidlc-state.md` 中重置所有受影响的步骤
   - 清除所有受影响计划文件中的复选框
   - 返回要重新启动的步骤
   - 从该点向前重新执行
5. **记录更改**：记录完整影响和重新启动原因

**考虑因素**：
- 需要大量返工
- 所有依赖步骤必须重做
- 时间线将显著延长
- 考虑修改是否比重新启动更好

---

### 5. 更改步骤深度

**场景**：用户想要更改当前或即将到来的步骤的深度级别

**示例**："让我们进行全面的需求分析而不是标准的"

**处理**：
1. **确认请求**："您想将需求分析从标准深度更改为全面深度。这将更彻底但需要更长时间。确认？"
2. **更新执行计划**：在 `workflow-planning.md` 中更改深度级别
3. **调整方法**：遵循步骤的全面深度指南
4. **更新估计**：通知用户新的时间线估计
5. **记录更改**：记录深度更改和原因

**考虑因素**：
- 更多深度 = 更多时间但更好的质量
- 更少深度 = 更快但可能遗漏细节
- 只能在步骤之前或期间更改，完成后不能更改

---

### 6. 暂停工作流

**场景**：用户需要暂停并稍后恢复

**示例**："我现在需要停下来，明天继续"

**处理**：
1. **完成当前步骤**：如果可能，完成正在进行的当前步骤
2. **更新复选框**：用 [x] 标记所有已完成的步骤
3. **更新状态**：确保 `aidlc-state.md` 反映当前状态
4. **记录暂停**：在 `audit.md` 中记录暂停点
5. **提供恢复说明**："当您返回时，我将检测您现有的项目并提供从以下位置继续：[当前阶段，当前步骤]"

**恢复时**：
1. **检测现有项目**：检查 `aidlc-state.md`
2. **加载上下文**：读取已完成步骤的所有产物
3. **显示状态**：显示当前步骤和下一步
4. **提供选项**：从上次停下的地方继续或审查之前的工作
5. **记录恢复**：在 `audit.md` 中记录恢复点

---

### 7. 更改架构决策

**场景**：用户想要从单体更改为微服务（或反之）

**示例**："实际上，让我们做微服务而不是单体"

**处理**：
1. **评估当前进度**：确定工作流进行到哪里
2. **解释影响**：
   - 如果在单元规划之前：影响最小，只需更新决策
   - 如果在单元规划之后：必须重做单元规划、单元生成、所有每单元设计
   - 如果在代码生成之后：需要大量返工
3. **推荐方法**：
   - 工作流早期：从应用设计步骤重新启动
   - 工作流后期：考虑修改是否可行 vs. 重新启动
4. **获得确认**：用户必须理解更改的完整范围
5. **执行更改**：遵循受影响步骤的重新启动程序

**考虑因素**：
- 架构更改具有级联效应
- 工作流早期 = 更容易更改
- 工作流后期 = 考虑成本 vs. 收益

---

### 8. 添加/删除单元

**场景**：用户想要在单元生成后添加或删除单元

**示例**："我们需要将支付单元拆分为支付和计费"

**处理**：
1. **评估影响**：确定哪些单元已完成设计/代码
2. **解释后果**：
   - 添加单元：需要为新单元进行完整设计和代码
   - 删除单元：需要将功能重新分配给其他单元
   - 拆分单元：需要为两个结果单元重做设计和代码
3. **更新单元产物**：
   - 修改 `unit-of-work.md`
   - 更新 `unit-of-work-dependency.md`
   - 修订 `unit-of-work-story-map.md`
4. **重置受影响的单元**：将受影响的单元标记为需要重新设计
5. **执行更改**：遵循受影响单元的正常单元设计和代码过程

**考虑因素**：
- 影响这些单元的所有下游步骤
- 如果依赖关系更改，可能影响其他单元
- 时间线影响取决于受影响的单元数量

## 处理更改的一般指南

### 在进行更改之前

1. **理解请求**：询问澄清问题，了解用户想要更改什么以及为什么
2. **评估影响**：识别所有受影响的步骤、产物和依赖项
3. **解释后果**：清楚地传达需要重做什么以及时间线影响
4. **提供替代方案**：有时修改比重新启动更好
5. **获得明确确认**：用户必须理解并接受影响

### 在更改期间

1. **归档现有工作**：在进行破坏性更改之前始终备份
2. **更新所有跟踪**：保持 `aidlc-state.md`、计划文件和 `audit.md` 同步
3. **沟通进度**：让用户了解正在发生的事情
4. **验证更改**：确保更改在所有产物中保持一致
5. **测试连续性**：验证更改后工作流可以顺利继续

### 在更改之后

1. **验证一致性**：检查所有产物是否与更改对齐
2. **更新文档**：确保所有引用都已更新
3. **完整记录**：在 `audit.md` 中记录完整的更改历史
4. **与用户确认**：验证更改是否满足用户的期望
5. **恢复工作流**：从新状态继续正常执行

## 更改请求决策树

```
用户请求更改
    |
    ├─ 是当前阶段吗？
    |   ├─ 是：可以修改或重新启动当前阶段
    |   └─ 否：转到下一个问题
    |
    ├─ 是已完成的阶段吗？
    |   ├─ 是：评估对依赖阶段的影响
    |   |   ├─ 低影响：修改并更新依赖项
    |   |   └─ 高影响：建议从该阶段重新启动
    |   └─ 否：转到下一个问题
    |
    ├─ 是添加跳过的阶段吗？
    |   ├─ 是：检查先决条件，添加到计划，执行
    |   └─ 否：转到下一个问题
    |
    ├─ 是跳过计划的阶段吗？
    |   ├─ 是：警告影响，获得确认，跳过
    |   └─ 否：转到下一个问题
    |
    └─ 是更改深度级别吗？
        ├─ 是：更新计划，调整方法
        └─ 否：与用户澄清请求
```

## 最佳实践

1. **始终确认**：在没有明确用户确认的情况下，永远不要进行破坏性更改
2. **解释影响**：用户需要在决定之前理解后果
3. **提供选项**：有时有多种方法来处理更改
4. **首先归档**：在进行破坏性更改之前始终备份
5. **更新一切**：保持所有跟踪文件同步
6. **彻底记录**：记录所有更改以进行审计跟踪
7. **之后验证**：确保工作流可以顺利继续
8. **灵活**：工作流应适应用户需求，而不是强制执行僵化的过程
